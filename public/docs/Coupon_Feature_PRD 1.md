## 1. Overview

This document defines the product requirements for the **Coupon & Discount Management** feature of the e-commerce platform. The feature enables administrators and operators to create, configure, and manage promotional coupons that can be applied by customers at checkout.

The system must support a wide range of discount strategies — from simple flat-amount codes to complex conditional promotions — while giving admins full control over eligibility rules, validity periods, and usage limits. This version also covers **home banner promotion of active coupons**, **referral-linked coupon codes** (tied to customer referral programs), and **isolated single-use codes** (unique codes generated per customer or campaign).

---

## 2. Goals & Objectives

### Primary Goal
Provide a flexible, admin-managed coupon engine that supports diverse promotional strategies, drives conversion, and allows precise targeting of customer segments and product scopes.

### Success Metrics

- Admin can create a fully configured coupon in under 3 minutes
- Zero checkout errors caused by coupon misconfiguration
- Coupon redemption rate tracked and reportable per campaign
- Support for at least 5 discount types out of the box
- 100% of coupon validations enforce all active conditions before applying
- Referral codes traceable back to the referring customer at all times
- Home banner coupon promotions configurable without developer involvement

## 4. Feature Areas

### 4.1 General Information

The foundation of every coupon is its general configuration block, which defines the code, purpose, validity window, and top-level discount value.

#### Coupon Code
- Unique alphanumeric code entered by the customer at checkout
- Required field; must be unique across all active coupons
- Admin-defined (manual entry) or system-generated (random)
- Case-insensitive matching at checkout
- Max length: 32 characters; allowed characters: A–Z, 0–9, hyphens

#### Coupon Code Type
Every coupon must be classified as one of three code types. This classification drives generation, distribution, and tracking behavior:

- **Standard** — a single shared code usable by any eligible customer (e.g., `SUMMER20`). Created manually or auto-generated by the admin.
- **Referral** — a code automatically generated and tied to a specific referring customer. When a referee applies the code at checkout, the system records the referral and can optionally reward the referrer. Each referrer receives a unique code (e.g., `REF-JANE42`).
- **Isolated / Single-Use** — a unique code generated per recipient, valid for one redemption only. Used for targeted campaigns, loyalty rewards, or one-time customer recovery. Codes can be generated in bulk and distributed via email.

#### Description
- Internal-facing text field for admin/operator reference
- Required; not shown to end customers
- Max length: 255 characters

#### Status
- **Enabled** — coupon is active and can be redeemed (default)
- **Disabled** — coupon exists but cannot be applied at checkout
- Status can be toggled at any time without deleting the coupon

#### Discount Amount
- Numeric input defining the value of the discount
- Interpretation (fixed amount vs. percentage) depends on the selected Discount Type
- Required when discount type is not "Buy X Get Y"

#### Validity Period
- **Start date** — date from which the coupon becomes redeemable (defaults to today)
- **End date** — date after which the coupon is no longer valid
- Both fields use a date picker; end date must be ≥ start date
- If no end date is set, the coupon does not expire

#### Free Shipping
- Boolean toggle (checkbox)
- When enabled, the coupon waives all shipping costs for the order regardless of other discount rules
- Can be combined with a discount amount or used standalone

| Field | Type | Required | Notes |
|---|---|---|---|
| Coupon Code | Text | ✅ | Unique, case-insensitive |
| Code Type | Radio (Standard / Referral / Isolated) | ✅ | Default: Standard |
| Description | Textarea | ✅ | Internal only |
| Status | Radio (Enabled/Disabled) | ✅ | Default: Enabled |
| Discount Amount | Number | Conditional | Required unless Buy X Get Y |
| Start Date | Date | ✅ | Default: today |
| End Date | Date | ❌ | Null = no expiry |
| Free Shipping | Checkbox | ❌ | Default: unchecked |

### 4.2 Discount Type

Defines how the discount amount is applied. Exactly one type must be selected per coupon.

| Discount Type | Description | Priority |
|---|---|---|
| **Fixed discount to entire order** | Deducts a fixed currency amount from the order total | Must Have |
| **Percentage discount to entire order** | Deducts a percentage of the order total | Must Have |
| **Fixed discount to specific products** | Deducts a fixed amount from eligible product line items only | Must Have |
| **Percentage discount to specific products** | Deducts a percentage from eligible product line items only | Must Have |
| **Buy X Get Y** | Customer buys a defined quantity (X) and receives another quantity (Y) free or discounted | Should Have |

#### Business Rules
- For "specific products" types, the admin must be able to select or search which products/categories the discount applies to
- For "Buy X Get Y", additional fields are shown: "Buy quantity (X)", "Get quantity (Y)", and optionally "Get product" (if Y differs from X)
- Discount amount field is hidden or replaced with dedicated fields when "Buy X Get Y" is selected

### 4.3 Referral Code Configuration

Referral coupons are auto-generated codes linked to an existing customer (the referrer). They enable word-of-mouth growth by rewarding both the referrer and the referee.

#### How It Works
1. Admin creates a Referral coupon template — defining the discount value, type, and validity
2. The system generates a unique code for each customer enrolled in the referral program (e.g., `REF-JANE42`, `REF-JOHN99`)
3. The referrer shares their personal code with friends
4. When a new customer (referee) applies the code at checkout, the system:
   - Applies the configured discount to the referee's order
   - Records the successful referral against the referrer's account
   - Optionally triggers a reward to the referrer (store credit, discount code, or notification)

#### Admin Configuration
- Define referral discount value and type (applies the same discount types as standard coupons)
- Set referee eligibility: new customers only (default) or any customer
- Enable/disable referrer reward and define reward type:
  - Fixed store credit amount
  - New isolated discount code issued to referrer
  - No automatic reward (manual processing)
- Set maximum referral reward per referrer (e.g., cap at 10 successful referrals)
- Bulk-enroll customers into the referral program or allow self-enrollment

#### Tracking
- Each code is permanently linked to a referrer customer ID
- Admin can view: referrer name/email, code, number of successful referrals, total discount issued, total rewards given
- Referral activity appears in the coupon reporting dashboard under a dedicated "Referral" tab

| Field | Type | Required | Default |
|---|---|---|---|
| Referee discount type & amount | Same as standard coupon | ✅ | — |
| Referee eligibility | Toggle (New customers only / All) | ❌ | New customers only |
| Referrer reward type | Select (None / Store credit / New code) | ❌ | None |
| Referrer reward value | Number | Conditional | — |
| Max referrals per referrer | Integer | ❌ | Unlimited |

### 4.4 Isolated / Single-Use Code Configuration

Isolated codes are unique, one-time-use codes generated in bulk or individually. Each code is valid for exactly one redemption and then automatically disabled. They are ideal for targeted campaigns, post-purchase recovery, loyalty gifts, or VIP promotions.

#### Generation
- Admin sets a **code prefix** (e.g., `VIP-`, `WELCOME-`) for brand consistency
- Admin sets the **quantity** of codes to generate (1 to 10,000 per batch)
- System appends a random suffix to ensure uniqueness (e.g., `VIP-X7T2Q`)
- All codes in a batch share the same discount configuration, validity dates, and conditions

#### Distribution
- **Download as CSV** — export all generated codes for use in email campaigns, CRM tools, or manual distribution
- **Send via email** — optionally assign codes to specific customer emails and trigger a system email with the code included
- Codes not yet distributed are visible in a "pending" state in the code list

#### Redemption Rules
- Each code can only be used once; it is automatically disabled after successful checkout
- If a code is used in an order that is later refunded/cancelled, admin can manually re-enable it
- Unused codes in a batch can be bulk-expired at any time

#### Management
- Admin can view the full list of codes within a batch: code, assigned email (if any), status (Pending / Used / Expired), redemption date
- Filter by batch, status, or assignment
- Bulk-expire unused codes from a batch

| Field | Type | Required | Default |
|---|---|---|---|
| Code prefix | Text | ❌ | None |
| Quantity to generate | Integer (1–10,000) | ✅ | 1 |
| Discount config | Same as standard coupon | ✅ | — |
| Distribution method | Select (Download CSV / Email / None) | ❌ | Download CSV |
| Auto-disable after use | Boolean | ✅ | Always on (locked) |

Rules that must be satisfied at the order level before the coupon can be applied.

#### Minimum Purchase Amount
- Numeric input (currency)
- The order subtotal must be equal to or greater than this value
- Default: 0 (no minimum)
- Tooltip helper text: "Set to 0 to apply with no minimum order value"

#### Minimum Purchase Quantity
- Integer input
- The total number of items in the cart must meet or exceed this value
- Default: 0 (no minimum)
- Tooltip helper text: "Set to 0 to apply regardless of item count"

#### Maximum Discount Cap *(recommended addition)*
- Optional numeric cap on the total discount applied, useful for percentage discounts
- E.g., "20% off, up to $50 maximum"

#### Usage Limits *(recommended addition)*
- **Total usage limit** — maximum number of times the coupon can be redeemed across all customers
- **Per-customer usage limit** — maximum redemptions per individual customer account

|---|---|---|---|

| Minimum purchase qty | Integer | ❌ | 0 |
| Maximum discount cap | Number (currency) | ❌ | None |

### 4.6 Customer Conditions

Rules that restrict coupon eligibility based on who the customer is.

#### Customer Groups
- Multi-select dropdown
- Coupon applies only to customers belonging to selected groups (e.g., VIP, Wholesale, New Customers)
- If no group is selected, coupon applies to all customer groups

#### Customer Emails
- Free-text or tag input for individual email addresses
- Coupon is restricted to customers whose account email matches one of the listed values
- Supports bulk paste (comma-separated)
- If empty, no email restriction is applied

#### New Customers Only *(recommended addition)*
- Boolean toggle
- When enabled, coupon can only be redeemed by customers with zero previous completed orders

| Field | Type | Required | Default |
|---|---|---|---|
| Customer groups | Multi-select | ❌ | All groups |
| Customer emails | Tag input | ❌ | No restriction |
| New customers only | Toggle | ❌ | Off |

### 4.7 Coupon List & Management

Admins must be able to manage existing coupons from a list view.

- **Columns:** Code, Description, Discount Type, Amount, Status, Start Date, End Date, Usage Count, Actions
- **Filtering:** by status (enabled/disabled), discount type, date range, customer group
- **Search:** by coupon code or description keyword
- **Bulk actions:** enable, disable, or delete multiple coupons at once
- **Inline status toggle:** enable/disable without opening the edit form
- **Duplicate coupon:** clone an existing coupon as a starting point for a new one
- **Pagination:** configurable rows per page (25 / 50 / 100)

### 4.8 Coupon Validation at Checkout

The checkout engine must validate all coupon conditions in real time when a customer applies a code.

#### Validation Sequence
1. Code exists and is not expired (start/end date check)
2. Status is Enabled
3. Customer is eligible (group and/or email match)
4. Order meets minimum purchase amount
5. Order meets minimum purchase quantity
6. Usage limits not exceeded (global and per-customer)
7. Applicable products are present in cart (for product-specific types)

#### Error Messaging
Each failed condition must produce a clear, customer-facing error:

| Condition Failed | Error Message |
|---|---|
| Code not found | "This coupon code is invalid." |
| Expired | "This coupon has expired." |
| Disabled | "This coupon is currently inactive." |
| Customer not eligible | "This coupon is not valid for your account." |
| Minimum amount not met | "Add \$X more to your cart to use this coupon." |
| Min qty not met | "Add X more items to use this coupon." |
| Usage limit reached | "This coupon has reached its usage limit." |

### 4.10 Home Banner — Coupon Promotion

Admins can configure a promotional banner displayed on the website's home page to highlight active coupon campaigns. This enables marketing without requiring developer changes and drives awareness directly at the point of highest traffic.

#### Banner Content
- **Headline text** — short promotional message (e.g., "Get 20% off your first order!")
- **Coupon code display** — visually highlight the code with a copy-to-clipboard button (e.g., `SUMMER20 📋`)
- **Supporting text** — optional subtitle with conditions summary (e.g., "Valid until March 31. Min. order $50.")
- **CTA button** — optional button with custom label and destination URL (e.g., "Shop Now" → `/products`)
- **Background** — solid color, gradient, or image from the asset bucket; color picker with live preview
- **Text color** — configurable to ensure contrast against the chosen background

#### Banner Behavior
- Admin selects which active coupon to promote from a dropdown (populated from enabled coupons)
- Banner is shown only while the linked coupon is within its valid date range; it auto-hides when the coupon expires
- Admin can manually show or hide the banner independently of the coupon's status
- Multiple banners can be created but only one is displayed at a time (admin sets priority/order)
- Banner supports a close/dismiss button for customers (session-based — re-appears on next visit)

#### Admin Controls
- Enable / Disable banner toggle
- Link to coupon (dropdown of active coupons)
- Set banner position: top of page (sticky bar) or hero section overlay
- Preview banner before publishing (WYSIWYG inline preview)
- Schedule banner: optional start and end datetime independent of the coupon's own dates

## 5. Non-Functional Requirements

- Coupon validation at checkout must complete in under 200ms
- All coupon configuration changes must be logged in the admin audit trail (user, timestamp, field changed)
- Concurrent redemption of the same coupon by multiple customers must respect usage limits without race conditions (use atomic counters or database locking)
- Coupon codes must be stored and matched case-insensitively
- Admin UI must be fully accessible (WCAG AA) and responsive on tablet and desktop
- Bulk operations (enable/disable/delete) must support batches of up to 500 coupons
- Isolated codes must be guaranteed globally unique at generation time; collisions must be retried automatically
- Referral codes must be permanently and immutably linked to their referrer customer ID; the link cannot be edited after generation
- Home banner must render within 100ms of page load and must not block the critical rendering path (load asynchronously)
- Banner auto-hide on coupon expiry must trigger within 5 minutes of the expiry datetime

## 6. Out of Scope (v1.0)

- Customer-facing coupon wallet or "My Coupons" page
- Automatic coupon application (no-code promotions applied without customer entry)
- Coupon stacking (applying multiple coupons to one order)
- A/B testing for coupon campaigns
- Loyalty points integration

## 7. Dependencies & Risks

### Dependencies
- Product catalog API must expose product/category lookup for product-specific discount types
- Customer group data must be available at checkout validation time
- Order service must expose subtotal and item count before payment processing

### Risks

| Risk | Mitigation |
|---|---|
| Race condition on usage limits | Use atomic DB increment with constraint check |
| Admin creates conflicting coupons | Add overlap detection warning on save |
| Performance impact on checkout | Cache active coupon rules; invalidate on update |
| Discount misapplication on product-specific types | Enforce product eligibility check server-side, not only client-side |
| Referral code abuse (self-referral) | Detect and block same account or same email domain registering as both referrer and referee |
| Isolated code generation collisions at scale | Use UUID v4 suffix + DB uniqueness constraint with retry loop |
| Banner blocking page render | Load banner component asynchronously; do not block LCP |
| Expired coupon still shown in banner | Implement server-side expiry check on every page load with a 5-min TTL cache |

| Phase | Timeline | Scope |
|---|---|---|
| **Phase 1** | Weeks 1–3 | General Info (code, description, status, dates, free shipping), Standard coupon codes, Fixed & Percentage discount to entire order, Coupon list view |

## 9. Open Questions

- Should coupon stacking be supported in a future version? If so, define priority rules.
- Should "Buy X Get Y" support cross-product promotions (buy product A, get product B free)?
- Is there a requirement for coupon codes to be importable via CSV bulk upload?
- Should expired coupons be auto-archived or remain visible in the list with an "Expired" badge?
- What is the audit retention policy for coupon usage logs?
- For referral codes: should the referrer be rewarded on the referee's first order only, or every order the referee places?
- Should isolated codes support a minimum purchase condition that overrides the batch-level setting per individual code?
- Can the home banner display multiple active coupons in a rotating carousel, or is it always a single promotion?
- Should the banner close/dismiss preference persist across sessions (requires authentication or persistent cookie)?
