# Decision Record: Wompi Payment Error Handling Enhancement

**ID:** DEC-0005  
**Date:** 2025-01-07  
**Status:** Completed  
**Type:** Enhancement  
**Priority:** High  

## Context

The Wompi payment system was experiencing poor error handling, particularly for validation errors like phone number format issues. When Wompi returned validation errors (e.g., "Los formatos de telefonos : 5142426933 no son válidos"), the system would:

- Close the payment modal completely
- Force users to re-enter all payment information
- Show generic error messages
- Provide poor user experience

## Problem Statement

Users were frustrated when validation errors occurred during payment processing because:

1. **Lost Work**: Payment modal closed on validation errors
2. **Poor Feedback**: Generic error messages didn't help users understand what to fix
3. **Re-entry Required**: Users had to start payment process over
4. **No Guidance**: No specific instructions on how to correct validation issues

## Decision

Implement intelligent error handling system that:

1. **Parses Wompi Error Responses**: Extract meaningful validation messages
2. **Categorizes Error Types**: Distinguish validation errors from system errors
3. **Smart Modal Behavior**: Keep form open for validation errors, close for system errors
4. **User-Friendly Messages**: Convert technical errors to actionable guidance
5. **Auto-Clear Functionality**: Remove errors when user starts correcting information

## Implementation

### Backend Changes (API Route)

```typescript
// Enhanced error parsing function
function extractUserFriendlyMessage(wompiError: any): string {
  if (wompiError.mensajes && Array.isArray(wompiError.mensajes)) {
    const message = wompiError.mensajes[0]
    
    // Phone number validation error
    if (message.includes("formatos de telefonos")) {
      return "Invalid phone number format. Please enter a valid El Salvador phone number without dashes (numbers only)."
    }
    
    // Card validation errors
    if (message.includes("tarjeta")) {
      return "Invalid card information. Please check your card details and try again."
    }
    
    // Email validation errors
    if (message.includes("email")) {
      return "Invalid email format. Please enter a valid email address."
    }
    
    return message
  }
  
  return "Payment processing failed. Please check your information and try again."
}

// Structured error response
const structuredError = {
  success: false,
  error: "validation_error",
  code: paymentResponse.status,
  wompiError: wompiError,
  userMessage: extractUserFriendlyMessage(wompiError),
  canRetry: true
}
```

### Frontend Changes (Payment Modal)

```typescript
// Smart error handling
if (responseData.error === "validation_error" && responseData.canRetry) {
  // Show validation error in form instead of closing modal
  setErrors({ general: responseData.userMessage })
  
  toast({
    title: "Validation Error",
    description: responseData.userMessage,
    variant: "destructive",
  })
  
  return // Stay in the form, don't close modal
}

// Auto-clear error functionality
const clearGeneralError = () => {
  if (errors.general) {
    const { general, ...rest } = errors
    setErrors(rest)
  }
}

// Applied to all form inputs
onChange={(e) => {
  clearGeneralError() // Clear error when user starts typing
  setCardData((prev) => ({ ...prev, field: e.target.value }))
}}
```

### UI Enhancements

## Results

### User Experience Improvements

1. **No Lost Work**: Validation errors keep payment form open
2. **Clear Guidance**: Specific instructions for fixing issues
   - Phone format: "Please enter a valid El Salvador phone number without dashes (numbers only)"
   - Card errors: "Please check your card details and try again"
   - Email errors: "Please enter a valid email address"
3. **Immediate Feedback**: Errors clear when user starts typing
4. **Reduced Frustration**: No need to re-enter payment information

### Technical Achievements

1. **Error Classification**: Validation vs system error categorization
2. **Type Safety**: All TypeScript errors resolved
3. **Structured Responses**: Consistent error format
4. **Extensible System**: Easy to add new error types

### Error Scenarios Handled

| Error Type | Wompi Response | User Message | Behavior |
|------------|----------------|--------------|----------|

| Email Format | "email" errors | "Invalid email format..." | Stay in form |

| API Failure | 500 errors | "Payment processing failed..." | Close modal |
