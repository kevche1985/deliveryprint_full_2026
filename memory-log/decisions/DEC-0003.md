## 🏗️ Architecture Evaluation

### **Option 1: Keep Separate Tables (RECOMMENDED)**

#### **Advantages:**
✅ **Clear separation of concerns**
- Physical products: Catalog items available to all users
- Digital products: User-specific AI-generated content

✅ **Different business logic**
- Physical: Inventory management, shipping, variants
- Digital: User ownership, generation parameters, download links

✅ **Different access patterns**
- Physical: Public browsing, no user ownership
- Digital: Private, user-specific access

✅ **Scalability**
- Independent scaling and optimization
- Different indexing strategies
- Separate backup/archival policies

✅ **Security**
- Different RLS policies for different use cases
- Easier to audit and manage permissions

#### **Current Schema Differences:**

**`products` table:**
```sql
- id (UUID)
- name (VARCHAR)
- description (TEXT)
- price (DECIMAL)
- category (VARCHAR)
- image (TEXT)
- is_active (BOOLEAN)
- is_featured (BOOLEAN)
- created_at (TIMESTAMP)
- updated_at (TIMESTAMP)
- SKU (VARCHAR)
```

**`digital_products` table:**
```sql
- id (UUID)
- user_id (UUID) -- Key difference: user ownership
- type (VARCHAR) -- logo, image, font
- name (VARCHAR)
- description (TEXT)
- file_data (JSONB) -- AI generation data
- generation_inputs (JSONB) -- User inputs
- base_price (DECIMAL)
- status (VARCHAR) -- unpurchased, purchased, etc.
- preview_url (TEXT)
- download_url (TEXT)
- metadata (JSONB)
- created_at (TIMESTAMP)
- updated_at (TIMESTAMP)
```

### **Option 2: Merge Tables (NOT RECOMMENDED)**

#### **Disadvantages:**
❌ **Complex schema**
- Many nullable columns for different product types
- Confusing business logic in single table

❌ **Performance issues**
- Larger table with mixed access patterns
- Complex queries with multiple conditions

❌ **Security complexity**
- Single RLS policy must handle multiple use cases
- Higher risk of data exposure

❌ **Maintenance overhead**
- More complex migrations
- Harder to optimize for specific use cases

## 🔐 Recommended RLS Policies

### **For `public.products` (Physical Products)**

#### **Current Issue:**
RLS likely enabled without policies, blocking all access.

#### **Recommended Policies:**

```sql
-- Enable RLS
ALTER TABLE products ENABLE ROW LEVEL SECURITY;

-- Policy 1: Public read access for active products
CREATE POLICY "public_can_view_active_products" ON products
    FOR SELECT 
    USING (is_active = true);

-- Policy 2: Admin management access
CREATE POLICY "admins_can_manage_products" ON products
    FOR ALL 
    USING (
        EXISTS (
            SELECT 1 FROM user_profiles 
            WHERE user_profiles.id = auth.uid() 
            AND user_profiles.role = 'admin'
        )
    );

-- Policy 3: Operators can view all products
CREATE POLICY "operators_can_view_all_products" ON products
    FOR SELECT 
    USING (
        EXISTS (
            SELECT 1 FROM user_profiles 
            WHERE user_profiles.id = auth.uid() 
            AND user_profiles.role IN ('admin', 'operator')
        )
    );
```

### **For `public.digital_products` (AI-Generated)**

#### **Current Status:**
Has user-specific RLS policies (correct approach).

#### **Recommended Improvements:**

```sql
-- Enable RLS (if not already enabled)
ALTER TABLE digital_products ENABLE ROW LEVEL SECURITY;

-- Policy 1: Users can manage their own digital products
CREATE POLICY "users_own_digital_products" ON digital_products
    FOR ALL 
    USING (auth.uid() = user_id)
    WITH CHECK (auth.uid() = user_id);

-- Policy 2: Admins can view all digital products
CREATE POLICY "admins_can_view_all_digital_products" ON digital_products
    FOR SELECT 
    USING (
        EXISTS (
            SELECT 1 FROM user_profiles 
            WHERE user_profiles.id = auth.uid() 
            AND user_profiles.role = 'admin'
        )
    );

-- Policy 3: Public can view purchased/published digital products
CREATE POLICY "public_can_view_published_digital_products" ON digital_products
    FOR SELECT 
    USING (
        status IN ('purchased', 'published') 
        AND metadata->>'is_public' = 'true'
    );
```

## 📈 Performance Considerations

### **Query Optimization:**

1. **Separate queries for different product types:**
```typescript
// Physical products
const getPhysicalProducts = () => {
  return supabase
    .from('products')
    .select('*')
    .eq('is_active', true)
    .order('created_at', { ascending: false });
};

// Digital products (user-specific)
const getUserDigitalProducts = (userId: string) => {
  return supabase
    .from('digital_products')
    .select('*')
    .eq('user_id', userId)
    .order('created_at', { ascending: false });
};
```

2. **Combined view for admin dashboard:**
```sql
CREATE VIEW admin_all_products AS
SELECT 
    id, name, price, 'physical' as product_type, 
    category, is_active, created_at
FROM products
UNION ALL
SELECT 
    id, name, base_price as price, 'digital' as product_type,
    type as category, (status != 'archived') as is_active, created_at
FROM digital_products;
```
